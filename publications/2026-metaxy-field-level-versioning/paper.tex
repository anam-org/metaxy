\begin{abstract}
Multimodal ML pipelines incur substantial GPU costs, yet existing orchestrators invalidate entire downstream datasets when any processing step changes---even for unaffected fields.
We present \textbf{Metaxy}, an open-source Python library providing \emph{field-level dependency tracking at record granularity}.
Feature definitions form a directed acyclic graph; for each record, hierarchical version hashes propagate changes along field-level edges.
A \texttt{resolve\_update} operation returns exactly those records that are new, stale, or orphaned, enabling selective recomputation.
In production across two organizations processing millions of multimodal samples, Metaxy eliminates redundant GPU work while preserving complete per-record lineage for reproducibility.
\end{abstract}

\section{Introduction}

Modern ML increasingly relies on multimodal data---video, audio, images, and text---processed through GPU-accelerated pipelines.
Unlike tabular ETL, where re-execution is cheap, multimodal steps such as speech-to-text transcription or embedding generation cost orders of magnitude more per record.
Data orchestrators like Dagster~\cite{dagster} model pipelines as graphs of \emph{assets} (tables or datasets) and track freshness at the asset level: when a processing step changes, every downstream record is recomputed---even those whose inputs are unchanged.

Consider a video pipeline extracting both audio transcripts and face detections (Figure~\ref{fig:field-deps}).
Improving only the audio denoising algorithm triggers recomputation of face detections that depend solely on video frames---a costly false positive.
This problem compounds as pipelines grow and is especially acute for iterative, agent-driven ML development, where rapid experimentation on feature pipelines demands that each iteration recomputes only what is strictly necessary.

We address this gap with \textbf{Metaxy},\footnote{\url{https://github.com/anam-org/metaxy}} a standalone metadata layer operating at \emph{field} and \emph{record} granularity that enables selective recomputation---both cost-efficient and supportive of rapid experimentation.

\subsection{Feature Definitions and Dependency Graph}

Feature definitions are registered through a common specification interface.\footnote{The most common authoring pattern uses Python classes extending a Pydantic base model, but the same graph can also be populated from \texttt{metaxy.lock} or directly from the metadata store.}
Each feature specifies identifier columns, computed fields with explicit code versions, and field-level dependencies on upstream features via a \texttt{FeatureSpec}.
At initialization, Metaxy constructs a global DAG where nodes are feature fields and edges encode data flow.
This graph enables topological version propagation: when an upstream field changes, only transitively dependent downstream fields are marked stale.
The system handles one-to-one, one-to-many, and many-to-one lineage relationships, covering common multimodal patterns such as expanding a video into per-frame crops or aggregating audio segments into a transcript summary.

\subsection{Hierarchical Versioning}

Version computation operates through deterministic hashing at multiple levels:
(1)~\emph{field code versions}---user-specified strings marking algorithmic changes;
(2)~\emph{field provenance}---per-field hashes combining current code versions with upstream record field versions;
(3)~\emph{record provenance}---a sample-level hash derived from the per-field provenance map.
Metaxy also exposes user-controlled data versions that default to provenance, allowing applications to encode direct data-version decisions when needed.
When \texttt{resolve\_update} is invoked, Metaxy compares expected provenance (computed from the current graph and upstream data) against stored provenance:

\begin{lstlisting}
inc = store.resolve_update("video/speech_to_text")
# inc.new:      records not yet processed
# inc.stale:    records with changed expected provenance
# inc.orphaned: records removed upstream
to_process = pl.concat([inc.new, inc.stale])
\end{lstlisting}

\noindent The returned \texttt{Increment} contains exactly those records requiring work; unchanged records are skipped entirely.

\subsection{Pluggable Storage, Orchestration, and Execution}

Metaxy separates metadata from storage, orchestration, and execution.
The \texttt{MetadataStore} abstraction supports multiple backends (e.g., DuckDB for local prototyping, ClickHouse or BigQuery or ducklake for production, or lakehouse formats like Delta Lake).
Version computation is pushed into the store via SQL where possible, avoiding data transfers.
Backend-agnostic dataframe interoperability is achieved through Narwhals~\cite{narwhals} and Ibis~\cite{ibis}.

For orchestration, Metaxy integrates with Dagster assets.
The \texttt{metaxify} decorator enriches asset definitions with lineage metadata while leaving scheduling decisions to the orchestrator.
For execution, Metaxy can be paired with Ray~\cite{ray} via dedicated datasource and datasink classes.
The orchestrator decides \emph{which feature} to compute; Metaxy resolves \emph{which samples actually require it}; the execution engine processes only the delta.

\section{Related Work}

DVC~\cite{dvc} versions datasets at file level without tracking individual records.
Asset orchestrators such as Dagster~\cite{dagster} schedule recomputation at asset granularity rather than tracking per-record, per-field provenance.
DataChain~\cite{datachain} supports delta processing for multimodal workflows but couples its incremental model to its own framework.
Metaxy fills this gap as a \emph{standalone} metadata layer providing field-level tracking at record granularity, decoupled from any specific compute or storage backend.

\section{Production Experience}

Metaxy has been in production at Anam\footnote{\url{https://www.anam.ai}}, a conversational AI avatar company, since December 2025, processing millions of training samples for a real-time interactive avatar platform powered by a custom video generation model.
The pipeline spans audio extraction, speech-to-text, face detection, video cropping, and embedding generation; the most expensive stages are GPU-accelerated.
Before Metaxy, changing crop resolution triggered recomputation of all downstream steps, including audio-only stages unaffected by the change.
With field-level tracking, only genuinely affected records are reprocessed.
The append-only metadata design preserves complete per-record lineage, enabling retrospective audits and experiment reproducibility without additional bookkeeping.

At ASCII (Austrian Supply Chain Intelligence Institute), Metaxy powers incremental document processing for knowledge extraction from large-scale web crawls~\cite{dagsterslurm}.

The same feature definitions can be reused across local prototyping and production deployments, reducing operational friction.

\section{Conclusion}

Metaxy introduces field-level dependency tracking at record granularity as a standalone metadata layer for multimodal ML pipelines.
By computing precise diffs before execution, it eliminates redundant GPU work while preserving complete lineage.
For the SAO community, Metaxy demonstrates that fine-grained provenance and cost-efficient incremental processing---properties essential for data systems supporting iterative and agentic ML workflows, feature pipeline iteration loops, and operational cost control---can be achieved through a lightweight layer that complements existing orchestrators rather than replacing them.
The library is open-source at \url{https://github.com/anam-org/metaxy}.
